# REQUIREMENTS.md — local-code-bench

> Product Requirements Document · v1 · 2026-06-20
> Generated by `/brainstorm`. Seed: `PROJECT-SEED.md`. Owner: FX.

## 1. Executive Summary

Local coding LLMs on Apple Silicon are now fast enough to drive agentic coding tools, but there is **no reproducible, apples-to-apples way to know which local model is actually best, whether it beats paying for GLM/Kimi K2 on OpenRouter, or how Codex performs as an agent baseline on the same tasks.** This project is a Python CLI harness that benchmarks coding models across one fixed machine (M3 Max 48GB) on **speed (TTFT/prefill, decode, latency), correctness (HumanEval/MBPP pass@1), and cost ($/task)**, while also running Codex CLI in a controlled agent mode. Success = committed results and leaderboards that answer "what should I run locally, is local worth it, and how does Codex compare as a coding agent?"

## 2. Context & Strategic Rationale

**Why now.** Two things converged in mid-2026: (1) 27B-class coding models quantized to 4-bit fit and run at interactive speed on a consumer Mac via MLX, and (2) coding-agent tools such as Codex can be run non-interactively enough to evaluate against repeatable task suites. The bottleneck is no longer *can you run it* but *which model or agent setup is worth using*.

**The thesis to test.** The two seed articles (Janardhan, *Data Science Collective*, Jun 2026) establish that **local agentic coding is prefill-bound, not decode-bound**: at a realistic 24k-token agentic context, a sparse MoE (Qwen3.6-35B-A3B) reaches first token in **31s vs 141s** for dense+speculative-decoding and **195s** for plain dense — a 4.5–6.3× swing on the metric that actually governs how an agent *feels*. Speculative decoding wins decode (145–182 tok/s) but loses the turn. **These claims deserve independent, reproducible measurement on this hardware — and extension to the cost question the articles never asked.**

**Competitive landscape.**
- **Published model cards / HumanEval leaderboards** — measure correctness, ignore local latency and cost-on-your-hardware. Not comparable to a 48GB Mac.
- **The seed articles** — measure speed rigorously but score **no correctness** and **no cost**, and run on 64GB. This project's differentiation: correctness + cost + the 48GB constraint, in a tool anyone can re-run.
- **SWE-bench / aider leaderboards** — gold-standard agentic correctness, but cloud-scale and not speed/cost-on-local-hardware oriented. (We defer SWE-bench-lite to a later phase rather than compete with it.)

## 3. User Problems & Jobs-to-be-Done

**Primary persona — "The local-AI tinkerer" (FX, and the public-repo audience like him).** A senior engineer running local models and Codex on a personal M-series Mac who wants the best coding experience and resents paying per-token for work a local model could do.

| Job-to-be-done | Current workaround | Why it sucks |
|----------------|--------------------|--------------|
| "Pick the fastest *usable* local model for my Mac" | Read blog posts run on different hardware (64GB ≠ 48GB); eyeball a demo video | Not reproducible, not on my hardware, no correctness signal — a fast model that writes broken code looks like a winner |
| "Decide if local is worth it vs. just paying OpenRouter" | Gut feel | No one has put local tok/s **and** cloud $/task on the same axis |
| "Re-check when a new model/quant drops" | Re-run ad-hoc manual tests, lose the old numbers | No standard suite, no stored results, no leaderboard to diff against |
| "Know whether Codex is worth using for this workload" | Try a few prompts by hand | No repeatable agent-mode baseline against the same suites |

**Friction map (today):** find a model → quantize/serve it (manual, MLX) → hand-drive a coding agent on a few prompts → form a vibe → forget the numbers → repeat from scratch next month. **The harness collapses the last steps into reproducible commands and persistent leaderboards.**

## 4. Solution Overview

**Approach.** A Python (`uv`) CLI with two benchmark modes. **Endpoint mode** treats every model as an **OpenAI-compatible `/v1/chat/completions` endpoint** and measures it under a controlled, single-turn protocol. The same harness hits a local MLX server and a remote OpenRouter base URL by swapping URL + key, so local and cloud land on one comparable axis. **Agent mode** runs Codex through `codex exec` in an isolated task workspace and scores the resulting code with the same benchmark tests where practical.

**Key capabilities (v1):**
1. Run a standard correctness suite (HumanEval, MBPP) against any subset of configured backends, unattended.
2. Capture per-task speed, token counts, pass/fail, and cost in raw JSONL.
3. Regenerate committed leaderboard sections ranking endpoint models by pass@1, latency, tok/s, and $/task, and Codex agent runs by pass@1 and wall time.
4. Execute generated code in an isolated sandbox; survive flaky/malformed backends without aborting the run.

**Technical architecture (overview).**
```
configs/models.yaml ──► endpoint runner ──► provider adapter (OpenAI-compatible | Anthropic baseline)
                          │                 └─► local MLX OR OpenRouter OR Claude API
                          ├─► metrics (stream → TTFT, prefill t/s, decode t/s, latency, tokens)
configs/agents.yaml ──► agent runner ───────► codex exec in isolated task workspace
                          │
                          ├─► task suite (HumanEval/MBPP loader) ──► sandbox runner (pass@1)
                          ├─► cost calc where reliable token data exists
                          └─► results/<run>.jsonl ──► LEADERBOARD.md generator
```

**What we are NOT building in v1 (scope fences):**
- ❌ Model training / fine-tuning / quantization — we consume pre-quantized models.
- ❌ GUI / web dashboard — CLI + committed Markdown/JSONL only.
- ❌ Auto-install/management of MLX servers or CCR — started manually; harness points at a running endpoint (a bring-up *script* is allowed, orchestration is not).
- ❌ Claude Code agentic loop (→ v2), SWE-bench-lite (→ later phase), LLM-judge quality scoring (→ v2), charts/plots (→ v2).
- ❌ Multi-machine / variable hardware — single fixed M3 Max 48GB.
- ❌ CI gating / regression alerts — run on demand.

## 5. Detailed Requirements

### P0 — Must-have (v1)
- **P0-1 Config-driven roster.** `configs/models.yaml` defines endpoint backends (name, type, base URL, model id, pinned revision, price/1k tokens). `configs/agents.yaml` defines Codex agent targets. Adding a model or agent target requires **no code change**.
- **P0-2 v1 roster (5 backends):** Local — Qwen3.6-27B 4-bit + DFlash (`dflash serve`); Qwen3.6-35B-A3B MoE (`turboquant-serve`, must fit 48GB). Cloud — GLM 4.6, Kimi K2 (OpenRouter). Baseline — Claude via Anthropic API.
- **P0-3 One-command run.** `uv run bench …` runs the full HumanEval (+MBPP) suite against any subset of endpoint backends, end to end, unattended.
- **P0-4 Per-task metric capture** → `results/<run>.jsonl`: prompt, raw response, **TTFT, prefill tok/s, decode tok/s, total latency, tokens in/out, pass/fail, $ cost** ($0 for local).
- **P0-5 pass@1 scoring** via the benchmark's provided unit tests, **temperature 0** where supported, deterministic.
- **P0-6 Sandbox execution.** Generated code runs in an isolated temp dir + subprocess with timeout; a model cannot wedge, pollute, or compromise the host.
- **P0-7 Fault tolerance.** A backend that times out, errors, or emits malformed output is logged, scored 0, and the run continues.
- **P0-8 `LEADERBOARD.md` generator.** Regenerates from `results/`: endpoint pass@1, median latency, prefill/decode tok/s, $/task per model, plus Codex agent pass@1 and wall-time summaries.
- **P0-9 Reproducibility metadata.** Each run records fixed seed/temp, pinned model revisions, suite version, timestamp, hardware tag.
- **P0-10 Codex agent mode.** `uv run bench --mode agent --agent codex ...` materializes each task in an isolated workspace, runs `codex exec` non-interactively with explicit sandbox settings, captures command metadata/output, scores the result, and records pass/fail + wall time.

### P1 — Should-have
- **P1-1 Sweep mode** — agentic-preamble padding (2k/8k/16k/24k tokens) to reproduce the articles' prefill-vs-context curve.
- **P1-2 Re-score without re-run** — score/leaderboard regenerate from stored JSONL offline.
- **P1-3 Backend subset & resume** — run/skip specific models; resume a partial run.

### P2 — Nice-to-have (v2+ candidates)
- **P2-1** Claude Code agentic loop (`ccr code -p`) end-to-end suite.
- **P2-2** SWE-bench-lite (its own phase — repo checkout + isolated env harness).
- **P2-3** LLM-as-judge quality scoring on top of pass/fail.
- **P2-4** matplotlib charts (TTFT-vs-context, pass@1 bars) as committed PNGs.

### Non-functional
- **Performance.** Harness overhead negligible vs. model latency; streaming-based TTFT measurement accurate to ~10ms.
- **Security.** Untrusted generated code runs sandboxed only; secrets (`OPENROUTER_API_KEY`, `ANTHROPIC_API_KEY`) via env/`.env`, never committed (gitleaks already in pre-commit).
- **Portability.** Pure OpenAI-compatible protocol; no hard dependency on a specific MLX server beyond the documented bring-up.
- **Quality.** TDD per FX standard — unit tests on metrics parsing, scoring, cost calc; README documents manual server bring-up.

## 6. Success Criteria & Metrics

**Leading indicators (does it work / get used).**
- A clean endpoint run across all 5 backends completes unattended and produces a `LEADERBOARD.md`.
- A Codex agent run over a bounded suite subset completes unattended and is represented in the leaderboard.
- Re-running the same config reproduces pass@1 within ±0 (deterministic) and latency within run-to-run variance.

**Lagging indicators (does it answer the question).**
- The leaderboard yields a defensible answer to: *"Which local model gives the best correctness-per-second on 48GB?"* and *"At what task volume does local beat OpenRouter on cost?"*
- Independently confirms-or-refutes the articles' prefill-bound / MoE-wins thesis on this hardware (via P1-1 sweep).

**Definition of "done" (v1 acceptance bar).**
1. `configs/models.yaml` defines all 5 endpoint backends; `configs/agents.yaml` defines Codex; additions need no code change.
2. `uv run bench …` runs full HumanEval (+MBPP) against any subset, unattended.
3. Per-task JSONL captures all P0-4 fields incl. cost.
4. `LEADERBOARD.md` regenerates with distinct endpoint and Codex agent sections.
5. Generated code is sandboxed (temp dir + subprocess timeout).
6. Harness survives a flaky backend (logged, scored 0, run continues).
7. Test coverage on harness logic; README documents server bring-up.
8. pass@1 at temperature 0; fixed seed + pinned revisions recorded per run.

## 7. Implementation Plan

**Phase 0 — Skeleton & protocol (P0-1, P0-4 core).** `uv` project, project rename to `local-code-bench`, `configs/models.yaml`, OpenAI-compatible provider adapter, streaming metrics (TTFT/prefill/decode/latency), single hardcoded prompt round-trip against one OpenRouter model. *Milestone: one model, one prompt, real metrics in JSONL.*

**Phase 1 — Correctness suite & sandbox (P0-3, P0-5, P0-6).** HumanEval/MBPP loader, sandboxed pass@1 runner. *Milestone: pass@1 scored for one backend on full HumanEval.*

**Phase 2 — Full matrix & resilience (P0-2, P0-7, P0-9).** Wire all 5 backends incl. Anthropic baseline special-case; fault tolerance; reproducibility metadata; cost calc (P0-4 cost). *Milestone: unattended run across all 5 backends survives a deliberately killed local server.*

**Phase 3 — Codex agent mode & leaderboard (P0-8, P0-10, P1-2).** Codex `codex exec` runner, agent-mode scoring, `LEADERBOARD.md` generator from stored JSONL, README bring-up guide. *Milestone: committed endpoint and Codex agent leaderboards, re-scorable offline where raw outputs permit.*

**Phase 4 (stretch, v1.x) — Sweep mode (P1-1, P1-3).** Prefill-vs-context curve; subset/resume.

**Dependencies:** local MLX servers (`dflash-mlx`, `turboquant-mlx`) installed and bringable-up manually; Codex CLI installed and authenticated for agent-mode runs; `OPENROUTER_API_KEY` and `ANTHROPIC_API_KEY` available. HumanEval/MBPP datasets fetchable.

**Resources:** solo (FX). No external services beyond OpenRouter/Anthropic billing.

## 8. Risks & Mitigation

| Risk | Type | Likelihood | Impact | Mitigation |
|------|------|-----------|--------|------------|
| **35B-A3B MoE doesn't fit 48GB** (articles ran 64GB) | Technical | Medium | High (drops a key contender) | Verify resident footprint early (Phase 2); fall back to a smaller MoE quant or smaller dense model; record the OOM as a finding |
| **HumanEval saturation** — top models all ~near-100%, no discrimination | Market | High | Medium | MBPP adds spread; correctness is a *floor* filter, ranking leans on speed+cost; SWE-bench-lite later for real discrimination |
| **Weak local models emit malformed/non-parseable code** | Technical | High | Low | P0-7 fault tolerance: score 0, continue; this *is* a signal, not a bug |
| **OpenRouter price drift breaks $/task** | Operational | Medium | Low | Prices in `models.yaml` data, dated; cost recomputable offline from token counts |
| **Endpoint protocol ≠ real agent experience** | Market | Medium | Medium | Codex agent mode is MVP; endpoint and agent results are kept distinct rather than over-normalized |
| **Untrusted generated code harms host** | Security | Low | High | Mandatory sandbox (P0-6); no network in sandbox; subprocess timeout |
| **Sandbox flakiness inflates failures** (timeouts, env drift) | Technical | Medium | Medium | Generous per-task timeout, pinned Python, retry-once-then-fail logged distinctly from model failure |
